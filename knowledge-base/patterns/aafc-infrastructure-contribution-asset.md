- Domain expertise accumulation

## Value Transfer Mechanisms

### 1. Pattern Extraction

**During infrastructure work, capture:**
- Decision rationale (why this approach?)
- Trade-offs evaluated (what alternatives considered?)
- Success criteria (how to measure good outcome?)
- Integration points (how components connect)
- Scaling considerations (what breaks at what size?)

**Examples:**
- "How AAFC handles large dataset processing"
- "Infrastructure scaling patterns for research data"
- "Tool selection criteria for scientific workflows"

### 2. Component Abstraction

**What can be generalized:**
- Data processing pipelines
- Workflow orchestration patterns
- Configuration management approaches
- Monitoring/observability systems
- Testing strategies for research software

**Boundary respect:**
- Domain-specific code stays at AAFC
- General patterns/templates extracted to meta-project
- No proprietary information leaves work context
- Attribution/credit given to origin

### 3. Process Templates

**Workflows that apply elsewhere:**
- Research project setup patterns
- Data validation procedures
- Quality assurance processes
- Documentation standards
- Collaboration protocols

### 4. Anti-Pattern Documentation

**What NOT to do (learned from experience):**
- Infrastructure choices that didn't scale
- Integration patterns that caused problems
- Optimization premature or insufficient
- Tool mismatches for use case

## Tracking Strategy

### During Infrastructure Work

**Weekly capture (5-10 minutes):**
- What infrastructure decisions made this week?
- What patterns emerged?
- What worked well? What didn't?
- What's generalizable?

**Document in:**
- `knowledge-base/patterns/` - Reusable patterns
- `knowledge-base/anti-patterns/` - Things to avoid
- `knowledge-base/decisions/` - Decision rationale

### After Major Milestones

**Pattern extraction (30-60 minutes):**
- Review infrastructure decisions
- Extract generalizable patterns
- Document trade-offs and rationale
- Create templates for reuse

**Update:**
- Meta-project pattern library
- Project-specific CLAUDE.md files
- Framework capabilities documentation

## Boundary Respect

### What CAN be shared:
✅ General architectural patterns
✅ Tool selection criteria
✅ Process improvements
✅ Performance optimization approaches
✅ Testing strategies
✅ Documentation patterns

### What CANNOT be shared:
❌ AAFC-specific data
❌ Proprietary algorithms
❌ Confidential research details
❌ Internal communications
❌ Budget/financial information

### Gray areas (ask first):
⚠️ Specific tool configurations
⚠️ Performance benchmarks
⚠️ Integration details
⚠️ Vendor evaluations
